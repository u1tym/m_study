# JWT からユーザー名を取得する技術仕様

## 1. 目的

HTTP リクエストに付随する **JWT（JSON Web Token）** を検証し、ペイロード内の **`username` クレーム**（または運用で定めた同一役割のクレーム）から **ログイン中ユーザーのユーザー名** を取得する。

本仕様は **Bearer ヘッダではなく HttpOnly Cookie に JWT を載せる方式** を想定する（モバイル／ブラウザで Cookie を送る構成）。

---

## 2. 前提条件

| 項目 | 説明 |
|------|------|
| トークン形式 | JWT（通常は **JWS**：ヘッダ・ペイロード・署名の 3 部分を `.` で連結） |
| 署名アルゴリズム | 例：**HS256**（共有秘密鍵）。運用で **RS256** 等の場合は検証方法が鍵ペアベースに変わる |
| 秘密鍵 | トークン発行側と **同一の** `SECRET_KEY`（または同等の鍵素材）を検証側が保持する |
| ユーザー名の格納場所 | JWT ペイロード（JSON）内のクレーム。**本システムでは `username` 文字列を使用**（別名の場合は発行仕様に合わせる） |
| トークンの運搬 | **Cookie**（例：名前 `access_token`）。**HttpOnly** のため JavaScript からは読めず、**HTTPS 経由でブラウザが自動送信**する |

---

## 3. Cookie から生トークン文字列を取得する

1. HTTP リクエストの **Cookie ヘッダ**を解析する。
2. 運用で決めた **Cookie 名**（例：`access_token`）に対応する値を取得する。
3. 値が **欠落**していれば「未ログイン」として扱い、以降の検証は行わない。

**注意:** 生トークンはログやエラーメッセージに **出力しない**（漏えい・セッション固定化のリスク）。

---

## 4. 署名検証（必須）

JWT は Base64URL デコード可能なため、**検証なしでペイロードだけ読むことは推奨しない**（改ざんトークンを信じる恐れ）。  
**必ず署名検証を行ってから**クレームを信頼する。

### 4.1 HS256 の場合（共有秘密鍵）

- **入力:** Cookie から取り出した JWT 文字列、事前共有の **秘密鍵**、アルゴリズム名（例：`HS256`）。
- **処理:** ライブラリ標準の **verify** 付きデコード（例：検証付き `decode`）を実行する。
- **失敗時:** 署名不一致・期限切れ・形式不正など → **ユーザー名は取得できない**（401 等はアプリポリシーに従う）。

### 4.2 検証に必要な環境変数（例）

| 名前 | 役割 |
|------|------|
| `SECRET_KEY` / `JWT_SECRET_KEY` | 署名検証用秘密鍵（発行サービスと同一） |
| `JWT_ALGORITHM` | 例：`HS256` |
| `COOKIE_NAME` | JWT を載せている Cookie 名（例：`access_token`） |

---

## 5. 検証済みペイロードからユーザー名を取り出す

1. 検証済みデコード結果は通常 **JSON オブジェクト（クレーム集合）**。
2. **ユーザー名**は、発行仕様で定めたキーを参照する（本系では **`username`**）。
3. 型・内容チェック（推奨）:
   - **存在する**
   - **文字列型**
   - **空でない**（トリム後も長さが 0 でない）

満たさない場合は「トークンは正しいがユーザー名として利用できない」として **401 または業務エラー**とする。

### 5.1 参考：よく併用するクレーム（例）

| クレーム | 例の意味 |
|----------|-----------|
| `sub` | ユーザー ID（文字列のことが多い） |
| `username` | ログイン ID／表示用ユーザー名 |
| `iat` | 発行時刻（UNIX 秒） |
| `exp` | 有効期限（UNIX 秒） |

ユーザー名取得の本命は **`username`**（運用が `sub` のみの場合は仕様を別途定義する）。

---

## 6. 実装パターン（フレームワーク非依存の手順）

どの言語でも、論理は次のパイプラインにまとめられる。

```
Cookie[名前] → 文字列トークン
    → 失敗なら「未認証」
トークン + 秘密鍵 + アルゴリズム → 署名検証付きデコード
    → 失敗なら「無効トークン」
ペイロード["username"] → 文字列検証
    → 成功時のみ「ユーザー名」確定
```

### 6.1 Web API（例：依存注入）

- **認証が必要なエンドポイント**では、上記パイプラインを **共通ミドルウェアまたは依存関数**にまとめる。
- ハンドラには **検証済みユーザー名（またはクレーム全体）**だけ渡し、**Cookie / 生 JWT は下位レイヤに広げない**。

### 6.2 ログ用途のみ検証しない場合（非推奨・限定利用）

デバッグで **検証なしデコード**する場合でも、**本番の認可判定には使わない**。かつログに **ペイロードやトークン全文を出さない**。

---

## 7. CORS・Cookie 連携（ブラウザ）

- **クロスオリジン**で Cookie を送る場合:
  - サーバ：**`Access-Control-Allow-Credentials: true`**
  - **許可オリジン**は具体指定（ワイルドカード `*` と credentials の組み合わせは不可になりがち）
  - クライアント：**`withCredentials: true`**（XHR / fetch）

Cookie が届かないと JWT も取得できず、ユーザー名も得られない。

---

## 8. エラー・境界条件一覧（設計時チェック）

| 状況 | 挙動の例 |
|------|-----------|
| Cookie なし | 未認証 |
| 秘密鍵未設定 | サーバ設定不備（503 等） |
| 署名検証失敗・期限切れ | 無効トークン（401） |
| `username` 欠落・空・型不正 | クレーム不備（401 等） |
| DB の `accounts.username` と突合 | JWT の `username` と DB を一致させてアカウント ID を引く、等は **別レイヤの業務ルール** |

---

## 9. セキュリティ要件（要点）

- 平文 JWT をログに残さない。
- **HTTPS** で Cookie を送る（`Secure` 属性と合わせる）。
- 検証なしデコードを認可に使わない。
- 秘密鍵をコードに直書きせず、環境変数やシークレット管理に置く。
